    JavaScript is a single-threaded, synchronous language. But why do most developers love JavaScript even though it is slower compared to some other programming languages? Sometimes people do not even consider it a traditional programming language, yet it remains extremely popular.

    In this article, we are going to see how JavaScript manages to handle both synchronous and asynchronous tasks. You may wonder, "But you said JavaScript is synchronous." By the end of this article, you will have the answer.

    We are not going to talk about the history of JavaScript or other related topics. We will focus only on how JavaScript works.

    JavaScript Engine

    The main part of JavaScript is the JavaScript Engine.

    We know that JavaScript code, like many other programming languages, is written in an English-like syntax. The JavaScript Engine first parses the code, breaks it into tokens, and converts it into an AST (Abstract Syntax Tree). It then compiles the AST and executes it.

    A common question is: Is JavaScript an interpreted language or a compiled language?

    A few years ago, the answer was simple: JavaScript was considered an interpreted language. However, modern JavaScript engines use a JIT (Just-In-Time) Compiler, which improves performance during execution.

    The JavaScript Engine starts compiling the code and then begins execution. While executing, it can optimize the code and recompile parts of it to improve performance.

    Execution Context

    To execute code, JavaScript uses something called an Execution Context.

    An Execution Context is an environment used to run a piece of code. JavaScript is a scripting language and can run in different environments.

    • Browser Environment
    • Node.js Environment

    Node.js allows JavaScript to run on the server side. In this article, we will mainly focus on the browser environment.

    An Execution Context is a container that holds everything required for code execution, such as variables, functions, and objects.

    There are mainly two types of Execution Contexts:

    1. Global Execution Context (GEC)
    2. Function Execution Context (FEC)
    ``` 
    console.log("Hello World");
    ```

    This code runs inside the Global Execution Context.

    Whenever a function is invoked, JavaScript creates a new Function Execution Context for it.

    Phases of Execution Context

    Every Execution Context has two phases:

    1. Memory Creation Phase
    2. Code Execution Phase

    Memory Creation Phase

    In this phase, JavaScript scans the entire code of the current context and allocates memory for variables and functions.

    • Variables declared with var are initialized with undefined.
    • Functions are stored with their complete function definitions.
    • Objects and other declarations are allocated memory.
    ``` 
    console.log(a);
var a = 10;
    ```

    This prints undefined because JavaScript performs memory allocation before execution.

    This behavior is commonly known as Hoisting.

    Code Execution Phase

    Once memory creation is complete, JavaScript starts executing the code line by line.

    To do this, JavaScript uses a Call Stack.

    Call Stack

    The Call Stack is used to manage execution contexts.

    1. The Global Execution Context is pushed into the Call Stack first.
    2. Function contexts are pushed when functions are invoked.
    3. Nested functions create additional execution contexts.
    4. The most recently added context executes first.
    5. All function contexts are eventually removed.
    6. The Global Execution Context is removed last.

    When the Call Stack becomes empty and there is nothing left to execute, the program execution is complete.

    Asynchronous JavaScript

    JavaScript itself is synchronous, but it runs inside environments such as browsers. The browser provides additional features that enable asynchronous behavior.

    • Web APIs
    • Callback Queue
    • Microtask Queue
    • Event Loop

    Event Loop

    The Event Loop continuously watches the Call Stack. Whenever the Call Stack becomes empty, it moves tasks from the queues into the Call Stack for execution.

    Web APIs

    • setTimeout
    • setInterval
    • DOM Events
    • addEventListener
    • Geolocation
    • Fetch API
    ``` 
    setTimeout(() => {
  console.log("Hello");
}, 5000);
    ```

    Callback Queue

    The Callback Queue follows the First In, First Out (FIFO) principle.

    Promises and the Microtask Queue

    ``` 
    fetch("api-url")
  .then((data) => {
    console.log(data);
  });
    ```

    When the Promise is fulfilled or rejected, the callback attached using .then() is placed into the Microtask Queue.

    Priority of Microtask Queue

    1. Execute synchronous code.
    2. Empty the Microtask Queue completely.
    3. Execute one callback from the Callback Queue.
    4. Check the Microtask Queue again.
    5. Repeat.

    Execution Priority

    1. Call Stack
    2. Microtask Queue
    3. Callback Queue

    The Event Loop continuously coordinates between these components.

    This is how the JavaScript environment enables asynchronous programming while JavaScript itself remains a single-threaded, synchronous language.
